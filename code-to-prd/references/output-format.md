# 输出格式

## 总原则

- PRD 主体以描述原模板功能为首要目标，优先回答“旧模板如何工作、用户会经历什么、功能为何能成立”。
- 文档拆分服务于更稳定地还原模板功能，不是为了把章节机械切碎。
- 先写模板级事实，再写场景级事实；只有当目标是交给生成系统时，才额外补生成附录。

## 输出层级

完整 PRD 建议分为 4 层：

1. 模板总层：模板入口、证据、全局数据、场景导航、共享依赖，以及必要时附加的生成约束入口。
2. 场景层：每个 `page-scene` 的功能、交互、UI、数据、状态与边界情况。
3. 场景内功能层：当某个场景内部还能拆出相对独立、可单独实现的功能块时，在该场景下继续拆分。
4. Overlay 层：每个 overlay 的挂靠关系、进入退出、回流逻辑、界面与交互。

默认先保证场景可独立还原；只有当单个场景内部仍包含可独立实现的主流程、窗口、面板、结算段或设置段时，才继续拆成功能层。

## 模板总层输出

单文件输出时，主入口文件可直接以 `# PRD` 或 `# PRD - <模板ID>` 为标题，并包含以下章节。

若采用多文件拆分模式，则职责应收紧为：

- `index.md`：模板目标、页面目标、核心结论、全局数据摘要、阅读顺序与子文件导航。
- `files.md`：证据清单、模板文件定位清单、共享依赖。
- `scenes.md`：场景总览、主链路、overlay 挂靠关系、场景与功能单元导航。

不要在 `index.md`、`files.md`、`scenes.md` 三处重复写同一批摘要正文。

单文件模式下可直接包含以下章节：

## 产品概述

简述模板用途、核心能力和主玩法。

## 页面目标

说明模板帮助用户完成什么任务，以及原模板要解决的教学或游戏目标。

## 核心结论

概括原模板最重要的结构、功能、状态机与实现关注点，并说明哪些结论已被源码证实。

## 证据清单

列出实际分析使用到的源码文件、布局文件、资源组。

## 模板文件定位清单

- 按“注册文件 / 布局文件 / 组件文件 / 代码文件 / 资源文件”分组。
- 至少包含：文件类型、具体路径、来源依据、作用、关联资源 / 被谁引用。
- 对布局或代码直接引用的图集资源，要同时列出 `png` / `json` 路径与主要图片名或状态用途。
- 这一节必须是查找结果，不只是“去哪里找”的方法说明。

## 全局数据摘要

- 给出模板运行所需的最小全局数据结构。
- 只保留跨场景共用的数据对象、全局状态、公共外部联动。
- 场景专属字段下沉到对应场景 `data.md`。

## 场景总览

- 列出全部 `page-scene`、`overlay-scene`、`substate`。
- 写清主场景切换链和 overlay 挂靠关系。
- 单场景模板也必须有这一节，只是场景数量更少。

## 共享依赖

列出共享能力、公共组件、共享资源或 `PRD/shared/` 文档入口，并说明哪些能力不应重复散写到单个场景中。

只有跨多个场景复用，或脱离单一宿主场景后仍可独立复用的能力，才提升到 `PRD/shared/`；若它只服务于一个宿主场景，应优先留在该场景目录或该场景下的 `features/`。

## 生成附录（仅在交给新系统时添加）

若目标是把 PRD 交给新系统生成模板，模板根层可额外补一个精简的生成附录。它不是模板事实主体，只负责收束生成阶段必需但无法直接写入事实 PRD 的约束。至少包含：

- 使用前提：生成前必须先读取哪些共享文档和模板子文件
- 必须保持的源码事实
- 默认生成策略：静态证据不足时的保守实现方式
- 禁止误猜项
- 人工校验点
- 完成判定

若确实需要单独的任务指令文件，可再补 `prompt.md`；否则可以直接并入生成附录，不必单独保留一个固定文件。

## 场景层输出

每个 `page-scene` 目录建议至少包含以下 5 个文件：

### `index.md`

- 场景职责摘要
- 进入条件 / 退出条件
- 与其它场景、overlay 的关系
- 本场景子文件导航

`scene-<sceneId>/index.md` 只保留该场景的高层摘要与导航，不重复展开完整组件清单、交互细节和数据字段，这些内容应分别下沉到 `ui.md`、`interaction.md`、`data.md`。

### `ui.md`

- UI 布局说明
- 视觉与资源映射
- 组件拆解

### `interaction.md`

- 交互逻辑
- 状态模型
- 交互流程
- 动画与反馈规则

### `logic.md`

- 功能逻辑
- 功能需求
- HTML 实现映射
- 边界情况
- 待确认项 / Assumptions

### `data.md`

- 本场景实际消费的数据字段
- 数据分支对逻辑和界面的影响
- 本场景相关外部联动

场景层重点是把“这个场景如何工作、用了什么资源、依赖哪些数据、用户能做什么”讲清楚，不要把模板总层信息整段重复下沉。

若该场景内部还包含可独立实现的功能块，应继续拆到 `features/<featureId>/`。判断标准至少满足 3 条：

- 有明确触发入口，而不是纯展示性碎片。
- 有相对完整的内部状态或流程，可单独描述输入、处理中状态、结束状态。
- 有明确输出结果、退出条件或对宿主场景的状态回流。
- 实现者可以在不通读整个模板的前提下，基于该功能文档完成该部分开发。

常见例子包括：主游戏流程、结算流程、设置面板、奖励面板、教程引导段。若它本质是独立弹层/窗口，并且有明确进入退出与回流关系，也可以直接归为 overlay，而不是普通 feature。

## 场景内功能层输出

每个功能单元建议放在 `scene-<sceneId>/features/<featureId>/` 下，至少包含：

### `index.md`

- 功能摘要
- 宿主场景
- 触发入口
- 完成条件 / 退出条件
- 与宿主场景、其它功能单元、overlay 的关系

### `logic.md`

- 功能规则
- 内部状态与状态迁移
- 约束、边界情况、待确认项

### `interaction.md`

- 用户可触发动作
- 事件链路
- 反馈与禁用规则

### `ui.md`（按需）

- 该功能独有的结构区块
- 资源映射
- 与场景主体 UI 的边界

### `data.md`（按需）

- 功能实际消费的数据字段
- 输入输出数据
- 对宿主场景或外部系统的影响

功能层不是把场景切碎，而是把“相对独立、可单独实现”的部分单独还原。若某块内容没有清晰边界，只应留在场景层，不要强行拆目录。

`feature` 只用于属于单一宿主场景的功能单元。若同一能力跨多个场景复用，或可以脱离宿主场景单独复用，应提升到 `PRD/shared/`，各场景只记录它的使用点、依赖时序和输入输出。

## Overlay 层输出

每个 overlay 应单独放在 `overlays/<overlayId>/` 下，至少包含：

### `index.md`

- overlay 摘要
- 挂靠的 `page-scene`
- 进入条件 / 退出条件
- 回流到哪个场景或状态

### `ui.md`

- overlay 的界面结构
- 视觉资源映射
- 关键组件说明

### `interaction.md`

- overlay 的交互逻辑
- 状态变化
- 动画和反馈

### `logic.md`

- overlay 的功能逻辑
- 约束、边界情况、待确认项

### `data.md`

- overlay 实际消费的数据字段
- 与宿主场景或外部系统的联动

注意：overlay 不是普通 page-scene 的平替。必须写清它挂靠谁、何时出现、何时退出、退出后把状态回流到哪里。

## `scenes.md` 输出要求

`scenes.md` 对单场景和多场景模板都必须存在，至少包含：

- `sceneId`、场景名、`sceneType`（`page-scene` / `overlay-scene` / `substate`）
- 场景职责摘要、主链位置、进入条件 / 退出条件摘要
- 是否属于主链路
- 场景切换主链
- overlay 与 page-scene 的挂靠关系
- 场景内部是否继续拆分成功能单元；若拆分，列出功能单元名、职责和与宿主场景的边界
- 生成约束（例如哪些必须是页面级场景，哪些只能是 overlay）

`scenes.md` 是导航总表，不承担场景正文。详细组件、交互规则、数据字段和实现细节应放到对应 `scene-<sceneId>/` 或 `features/` 目录中。

判断标准不是“场景名列出来了”，而是“另一个工程师或 AI 能不能按模板总层 + 场景层 + overlay 层独立搭建页面宿主、overlay 栈和状态回流”。

## 多文件拆分模式（按需载入）

当 PRD 内容超过 300 行，或用户明确要求“按需载入”时，优先按场景拆分，而不是先按全模板的 UI / 交互 / 数据 / 实现四大类平铺拆分。单场景和多场景模板统一采用同一目录模型：

```text
PRD/<模板ID>/
	index.md                 — 模板入口；说明读取顺序、模板目标、核心结论、全局数据摘要、场景导航（每次必读）
	files.md                 — 模板级证据清单、源码/皮肤/资源定位、共享依赖
	scenes.md                — 所有模板必有；列出 page-scene、overlay-scene、substate、主切换链与功能单元导航
	scene-<sceneId>/
		index.md               — 场景摘要、职责、与其它场景/overlay 的关系
		ui.md                  — 本场景 UI 布局说明、视觉与资源映射、组件拆解
		interaction.md         — 本场景交互逻辑、状态模型、交互流程、动画规则
		logic.md               — 本场景功能逻辑、功能需求、边界情况、待确认项
		data.md                — 本场景实际消费的数据字段、分支影响、外部联动
		features/
			<featureId>/
				index.md         — 功能摘要、宿主场景、触发入口、完成/退出条件
				logic.md         — 功能规则、内部状态、边界情况、待确认项
				interaction.md   — 功能交互逻辑、事件链、反馈规则
				ui.md            — 功能独有 UI 与资源映射（按需）
				data.md          — 功能实际消费的数据字段、输入输出、联动（按需）
	overlays/
		<overlayId>/
			index.md             — overlay 摘要、挂靠 page-scene、进入/退出/回流规则
			ui.md                — overlay 的界面结构与资源映射
			interaction.md       — overlay 的交互、状态和反馈规则
			logic.md             — overlay 的功能逻辑、约束、边界情况、待确认项
			data.md              — overlay 实际消费的数据字段与联动

PRD/shared/<公共能力名>/
	index.md                 — 共享能力入口（存在共享组件时才创建）
	*.md                     — 公共组件职责、资源契约、生命周期、迁移约束、兼容性说明
```

**拆分规则：**

- `index.md` 必须是整套 PRD 的唯一入口，写清模板级文件、场景目录、overlay 目录、读取顺序和跳过风险。
- 多文件模式下，`index.md` 负责入口和阅读顺序，`files.md` 负责证据与定位，`scenes.md` 负责导航与主链；不要把同一段模板摘要同时写进这 3 个文件。
- `scenes.md` 对单场景和多场景模板都必须存在；单场景模板至少也要保留一个主场景目录，例如 `scene-main/`，不要另走一套简化结构。
- 场景目录中的 `logic.md` / `interaction.md` / `ui.md` / `data.md` 必须先描述原模板功能，再谈生成映射；若场景内部存在相对独立功能块，应拆到 `features/`，并写清宿主场景、触发条件、内部状态、输出结果和状态回流。
- `overlay` 必须单独列目录，并写清挂靠的 `page-scene`、进入/退出条件和状态回流；不要把真正的 overlay 混写成普通 feature，也不要把纯场景内逻辑片段误拆为 overlay。
- 模板级事实必须保留在模板根层，跨场景主链必须写在 `scenes.md`；共享能力优先抽到 `PRD/shared/`，全局通用数据结构也应在模板根层保留摘要或入口。`feature` 只属于单一宿主场景，跨场景可复用能力应提升为 shared。
- 每个原始事实章节只归入一个事实子文件，不得跨文件重复；仅当目标是“把 PRD 交给新系统生成模板”时，才额外附加生成附录，且它只能补充实现约束，不能取代事实文档。

## 输出约束

- `serialize/unserialize` 是框架层的多人课堂实时同步机制，不属于单页面交互逻辑，PRD 中不输出。
- 正常数据分支写入数据影响说明，不要散落到边界情况。
- 原始框架坐标仅用于推断区块相对关系，不得直接变成 HTML 绝对定位事实。
- 场景层、功能层和 overlay 层必须先把原模板功能讲清楚，再写 HTML 映射与生成约束。
- 模板级事实、跨场景主链、共享依赖，不能只写在单个场景文件里。
- 功能单元必须能够脱离整份 PRD 被单独理解：至少要能回答“它由谁触发、依赖什么状态、内部怎么运转、结束后把什么结果回给谁”。
- 若最终目标不是直接投喂生成系统，可完全省略生成附录。
