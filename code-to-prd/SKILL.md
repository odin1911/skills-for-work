# 代码转 PRD 技能

将现有页面代码、布局文件、组件、皮肤和资源事实转化为结构化、可复用的分析文档。

## 目标
输出结果必须足够具体，使另一个工程师或 AI 能在较低歧义下理解页面结构、核心交互、状态变化与数据契约，并据此完成页面重建、逻辑复刻或后续重构。

如果信息存在缺口，必须直接指出缺失规格面，不要模糊概括。

---

## Workflow

### 1. 整理证据范围
- 列出本次分析实际使用到的源码文件。
- 将文件按以下类别拆分：逻辑文件、布局文件、组件文件、皮肤/样式文件、资源清单文件。
- 如果某个行为来自公共框架而不是模板本地代码，必须标记为 `Assumption`。
- 若是 Egret/EUI 教学模版，还应定位模版 key、目录、入口文件、皮肤文件、template.json 与资源清单。

#### 模板文件快速定位规则
- 先在 `tools/template.json` 中根据模板 key 查找 `name` 与 `skin`，确认代码目录名和资源目录名；模板 key、代码目录名、皮肤目录名可能并不完全一致，禁止直接猜测目录。
- 代码文件优先在 `src/template/<name>/` 查找，至少覆盖：`Main.ts`、`SceneContainer.ts`、`controller/*.ts`、`component/*.ts`、`common/*.ts`。
- 布局文件优先在 `resource/assets/template/<skin>/eui_skin/` 查找；主皮肤一般为主场景 exml，组件皮肤通常在 `eui_skin/ccomponents/*.exml` 下。
- 组件与皮肤的绑定关系需在 `resource/default.thm.json` 中反查，明确“哪个类对应哪个 exml”。
- 模板资源组、图集注册关系需在 `resource/default.res.json` 中确认；默认只列标准 `1x` 资源路径。
- 若 exml 或代码中出现 `Common_5_json.xxx`、`Common_1_json.xxx` 或 `<模板>_json.xxx` 之类引用，必须继续追溯到对应图集的 `.json` 和 `.png` 文件，并说明该资源用于什么组件或状态。
- 若存在 `DataConst.ts`、`CommonDefine.ts` 等常量文件，应优先记录其中的布局常量；若不存在，则从 controller / component 文件中提取尺寸、gap、offset 和布局计算公式，禁止自行计算。
- 输出时必须把“模板相关文件的位置 + 每个文件的作用”写清楚，便于后续溯源时快速定位。

### 2. 分析 UI 布局与视觉资源
- 识别整体页面结构，例如：头部、底部、侧边栏、主内容区。
- 提取组件层级关系。
- 推断布局方式，例如：坐标布局、栅格、流式、分页容器、对齐方式、间距规则。
- 如果存在明确的坐标、尺寸、锚点、缩放信息，仅作为参考记录区块间的相对位置关系（左右、上下、主副、嵌套），不得作为 HTML 定位事实直接使用；HTML 实现应采用 flex / grid 等常见 Web 布局方式还原区块关系。
- 区分页级容器与可复用组件。
- 如果是 Egret / EUI 这类非 DOM 布局，也要翻译为 HTML 可理解的结构分区。
- 识别每个组件依赖的视觉资源。
- 提取图片、动画、音频、皮肤、纹理集、字体等依赖。
- 区分哪些资源是核心必需，哪些资源是辅助展示。
- 如果资源命名或资源组织方式是框架专有的，必须说明它们与页面结构或组件职责的映射关系。

### 3. 分析功能、交互与状态
- 从代码中归纳业务逻辑，不得假设代码中未出现的功能。
- 识别所有用户交互，包括点击、输入、切页、播放、切换模式等，并明确事件链路：触发条件 -> 处理器/控制器 -> 状态变化 -> UI 更新。
- 提取动态行为，包括条件渲染、事件派发、资源切换、外部系统调用。
- 列出关键状态：初始态、作答中、错误待重试、正确/揭晓态、完成态、重置态、恢复态。
- 写清状态迁移条件、约束关系，以及不同状态下的 UI 表现。
- 记录动画、持续时间、输入禁用时间窗、视觉反馈状态、可点击/不可点击状态。
- 如果是教学/答题模版，重点说明：单题/分页/多轮结构、判题时机、retry / reveal / reset / result 分支。序列化机制属于框架层多人同步，不分析（见规则）。

### 4. 提取数据契约
- 识别渲染页面所需的最小数据结构。
- 列出每个渲染对象的必填字段。
- 提取派生规则，例如排序、分页、过滤、资源选择、默认值策略。
- 记录数量限制；数据取值对逻辑和界面的分支影响（如题目数量、资源字段有无）应写入"题目数据影响说明"章节，而非散落在功能逻辑或边界情况中。
- 如果页面接入外部系统，必须记录触发时机、最小 payload 结构、必要字段和可选字段。
- 必须明确哪些字段是页面正确运行时不能缺失的。

### 5. 局部更新已有子文件
若用户指定更新某个已有子文件（如"更新 ui.md"、"重新提取 interaction.md"），进入局部分析模式（详见"局部分析模式"章节）：只读取该子文件对应的源文件类型，不重新读取无关文件，更新完成后输出变更摘要。

### 6. 评估 PRD 是否足以驱动生成
当用户目标不是“只分析代码”，而是“把 PRD 交给新系统生成模板”时，必须额外检查下面 5 件事：

- 若用户已经提供现有 PRD，并要求审查“是否足够生成”，应先读取 `index.md` 和相关 PRD 子文件，再只对存在争议的点最小回源到源码；不要重新做一遍全量模板扫描。
- `index.md` 是否已被定义为唯一入口，并写明读取顺序、文件职责和跳过风险。
- 若模板属于多场景，是否已经补出独立 `scenes.md`，并明确区分 `page-scene`、`overlay-scene`、`substate`。
- 若模板依赖公共能力（如 DragonBones、公共弹窗、公共状态容器），是否已把共享能力抽到 `PRD/shared/<能力名>/`，模板层只保留使用点、时序和依赖关系。
- 高优先级待确认项里，哪些必须继续保留为人工校验，哪些可以下沉为 `generation.md` 中的默认生成策略或显式 TODO。
- 是否已经存在可直接投喂生成系统的说明词，并明确“生成模板”而不是“继续分析 PRD”。

若上述任一项缺失，不能只回答“PRD 基本够用”，而要直接补文件或补规则。

### 7. 生成输出
- 按结构化 PRD 章节顺序用中文输出。
- 如偏向逻辑分析，强化业务规则、事件链和状态迁移。
- 如偏向页面重建，补充结构映射与实现约束。
- 必须先按“模板文件快速定位规则”完成查找，再把实际查找到的结果写入 PRD；不要只写查找方法，还要落出“文件路径 + 文件作用 + 关联资源 + 来源依据”。
- PRD 中必须输出“模板相关文件的具体路径 + 每个文件的作用”，做到后续查看时可直接按路径定位。
- 对布局文件或代码文件中引用到的资源文件，也要继续列出其路径和用途，尤其是图集 `.png` / `.json` 与主要图片名。
- 若输出目标包含“交给新系统生成模板”，除模板事实 PRD 外，还应补齐 `scenes.md`、`generation.md`、`prompt.md` 或共享能力文档；不能只停在基础 6 文件。

## 输出格式

# PRD

## 产品概述
简述页面作用与核心能力。

## 页面目标
说明该页面帮助用户完成什么任务。

## 核心结论
概括该页面最重要的结构、功能与实现关注点，并指出当前证据能支持确认到什么程度。

## 证据清单
列出实际分析使用的源码文件、布局文件、资源组。

## 模板文件定位清单
按“注册文件 / 布局文件 / 组件文件 / 代码文件 / 资源文件”分组列出具体路径，并说明每个文件的作用。
建议优先使用表格输出，至少包含：文件类型、具体路径、来源依据、作用、关联资源 / 被谁引用。
写法应尽量清晰到可直接复用：不仅列主文件，还要说明对应资源文件的路径、用途，以及它们是被哪个布局或代码文件使用的。
这一节必须是**查找结果**的输出，而不只是“去哪里找”的说明。
对布局或代码中直接引用的图集资源，需同时列出对应的 `png` 与 `json` 路径，以及主要图片名或状态用途，便于后续查看样式。

## 最小数据契约
给出渲染和驱动页面所需的最小 JSON / 数据结构，只列必填字段和字段说明，不描述数据的行为影响。

## 题目数据影响说明
列出题目数据的不同取值对模板行为和界面的全部影响，供开发人员和 AI 助手覆盖所有可能的情况。
以表格形式输出，列为：数据维度 | 取值/情况 | 影响（逻辑+界面）。
覆盖维度至少包括：核心 id 字段的合法性校验、题目数量分支、各类资源字段缺失时的降级行为，以及影响布局切换的数量/类型分支。
逻辑影响和界面影响写在同一行，不强制分开。

## UI 布局说明
描述结构分区、层级、定位关系、视觉重心。

## 视觉与资源映射
将组件与图片、动画、音频、字体、皮肤资源建立映射关系。

## 组件拆解
列出组件及其职责。

## 功能逻辑
写清系统如何运转：主要业务流程、条件分支、判题或切换规则。

## 交互逻辑
写清用户做什么会触发什么：可点击区域、输入入口、事件链与互斥限制。

## 状态模型
写清有哪些状态、如何切换，以及各状态下的界面表现与约束关系。

## 功能需求
将关键规则、限制条件、实现约束整理为可执行条目，避免与流程描述重复。

## 交互流程
用时序步骤串起初始化、选择、翻页、播放、上报、恢复等主路径。

## 动画与反馈规则
写明时长、顺序、禁用输入时窗、高亮反馈、空状态等规则。

## 外部联动
列出事件、接口、埋点、答题系统、上报结构与触发时机。
**注意**：`serialize/unserialize` 是框架层的多人课堂实时同步机制（Master 将当前场景状态序列化后通过网络推送给 Student 端恢复），不属于单页面交互逻辑，**PRD 中不输出此部分**，直接移除。

## HTML 实现映射
将页面拆解为 HTML 可直接实现的说明，至少包含：
- 建议的 DOM 区块划分
- 每个区块需要的关键 class / state
- 需要的前端状态变量
- 需要的主要事件绑定
- **布局方案**：说明各区块采用的 flex / grid 容器结构，指明主轴方向、对齐方式、间距基准；不得使用 `position: absolute` 复制原始坐标

## 边界情况
列出超出预期输入、运行时异常和未定义行为，例如：数量越界、资源加载失败、拖出画布边界等。正常数据范围内的分支行为（题目数量、资源字段有无）应写入"题目数据影响说明"，不在此重复。

## 待确认项 / Assumptions
列出实现时必须补齐、但当前代码无法严格证明的信息。每条标注优先级：
- **高**：影响页面功能或交互正确性，需人工在真实环境验证后补入 PRD。
- **低**：框架层细节或不影响单页面行为，可标注"不适用"后跳过。

## 场景功能清单（多场景模板必需）
若模板是多场景结构，必须额外输出独立的“场景功能清单”文档或章节。至少包含：
- `sceneId`、场景名、`sceneType`（`page-scene` / `overlay-scene` / `substate`）
- 进入条件、退出条件、核心功能、主要组件、允许交互、依赖的公共能力
- 生成要求、人工校验项、是否属于主链路
- 场景切换主链
- 生成约束（例如：哪些必须是页面级场景，哪些只能是 overlay，哪些只能留在内部状态机）

判断标准不是“场景名列出来了”，而是“另一个工程师或 AI 能不能按场景独立搭建页面宿主、overlay 栈和状态回流”。

## 生成补丁（交给新系统时必需）
若 PRD 的最终用途是交给新系统生成模板，必须补独立 `generation.md` 或等价章节，至少包含：
- 使用前提：生成前必须先读取哪些共享文档和模板子文件
- 必须保持的源码事实
- 默认生成策略：静态证据不足时的保守实现方式
- 必须生成的实现模块
- 禁止误猜项
- 人工校验点
- 交付判定

这部分不是源码事实清单，而是把“静态证据不足但首版必须落地”的内容收束成统一生成规则。

## 投喂说明词（交给新系统时必需）
若用户要把整套 PRD 直接交给生成系统，必须补独立 `prompt.md` 或等价章节，明确：
- 入口文件和强制读取顺序
- 任务目标是“生成可运行首版模板”，不是“继续分析旧项目”
- 必须生成的模块和场景骨架
- 默认策略、禁止误猜项和完成判定

`prompt.md` 是任务指令文件，不是新的模板事实源，不能替代 `files.md`、`data.md`、`ui.md`、`scenes.md`、`interaction.md`、`impl.md`。

## 多文件拆分模式（按需载入）

当 PRD 内容超过 300 行，或用户明确要求"按需载入"时，将单文件 PRD 拆分为“基础 6 个子文件 + 按需扩展文件”：

```
PRD/<模板ID>/
  index.md        — 产品概述、页面目标、核心结论、子文件导航表（每次必读）
  files.md        — 证据清单、模板文件定位清单（定位源文件/皮肤路径时读）
  data.md         — 最小数据契约、题目数据影响说明、外部联动（创建/调试模板数据时读）
  ui.md           — UI 布局说明、视觉与资源映射、组件拆解（修复视觉/资源问题时读）
  interaction.md  — 交互逻辑、状态模型、动画规则、交互流程（修复交互/状态机问题时读）
  impl.md         — 功能逻辑、功能需求、HTML 映射、边界情况、待确认项（生成 HTML/调试逻辑时读）
  scenes.md       — 多场景模板的场景功能清单（多场景模板必加）
  generation.md   — 面向新系统生成的默认策略、禁止误猜项、交付判定（要交给新系统时必加）
  prompt.md       — 交给新系统的投喂说明词（要直接投喂时必加）

PRD/shared/<公共能力名>/
  index.md        — 共享能力入口（存在共享组件时才创建）
  *.md            — 公共组件职责、资源契约、生命周期、迁移约束、兼容性说明
```

**规则：**
- `index.md` 必须包含子文件导航表，说明每个文件的内容、使用场景、读取顺序，并明确它是整套 PRD 的唯一入口。
- `scenes.md` 只在模板属于多场景时创建；若模板是多场景却省略它，视为 PRD 不完整。
- `generation.md` 与 `prompt.md` 只在目标是“把 PRD 交给新系统生成模板”时创建，但一旦进入该目标，它们就不是可选附件。
- 若某项能力被多个模板共享，应优先抽到 `PRD/shared/` 下；模板目录只保留该能力在本模板中的使用点、时序和依赖关系。
- 每个原始事实章节只归入一个事实子文件，不得跨文件重复；`generation.md` 与 `prompt.md` 可以总结实现约束，但不能反向取代事实文档。

## 局部分析模式

当用户指定更新多文件拆分 PRD 中的某个子文件时，只读取该子文件对应的源文件类型，不重新读取无关文件：

| 子文件 | 应读取的源文件类型 |
|--------|-------------------|
| `ui.md` | EXML 皮肤文件 + `default.thm.json` |
| `data.md` | controller / data 相关代码（如 `DataManager.ts`、`DataConst.ts`） |
| `interaction.md` | controller + component 文件 |
| `files.md` | `tools/template.json` + `default.thm.json` + `default.res.json` |
| `impl.md` | controller + component 完整逻辑文件 |
| `scenes.md` | 主容器 / scene 管理文件 + controller / component + 场景宿主相关皮肤文件 |
| `generation.md` | 现有 `scenes.md` + `interaction.md` + `impl.md` + 待确认项；必要时补读对应公共组件源码或共享文档 |
| `prompt.md` | `index.md` + `generation.md` + 其引用的子文件导航，不直接重读无关源码 |
| `index.md` | 不需重新读源文件；从其他子文件中提炼摘要 |

**规则：**
- 局部模式只输出目标子文件的变更内容，不重新生成整个 PRD。
- 更新完成后输出一行变更摘要，说明修改了哪些章节及原因。
- 若发现读取结果与现有子文件内容一致，明确告知"内容无变化，无需更新"。
- **必须逐字段/逐属性细粒度对比**，而不是概要印象对比：坐标、尺寸、可见性、资源 key、叠放顺序、状态默认值等每一项都需逐一核查。漏掉细节（如 `visible=false`、`width.down="85%"`、layer 顺序）等同于对比失败。

## 规则
- 若用户需要溯源、复刻或后续持续分析，必须完整描述模板相关文件的位置和作用，使后续流程可以快速定位到对应代码、皮肤和资源。
- PRD 产物中必须显式输出这些文件路径与作用说明，不能只在分析过程里隐含使用而不写出来。
- 默认只列模板实际使用到的标准 `1x` 资源路径。
- 严格基于代码与资源事实输出。
- 不得臆造功能。
- 不确定的内容必须标记为 `Assumption`。
- 保持简洁，但不能牺牲可实施性。
- 优先写实现事实，不要只写概述性语言。
- 每条关键需求都应能被翻译为结构、状态和行为要求。
- 不仅要说明“页面做什么”，还要说明“实现者必须构建什么”。
- 对布局文件或代码文件中直接使用到的资源文件，必须一起列出路径和用途。
- 如果项目中不存在独立常量文件，要明确说明布局常量分散在哪些 controller / component 中。
- 如果某部分无法从现有代码充分推断，必须直接指出缺口，而不是模糊描述。
- `serialize/unserialize` 是多人课堂实时同步的框架机制（Master→Student 网络推送），单页面交互 PRD 不分析、不输出此部分。
- 待确认项必须标注优先级（高/低）：高优先级需人工验证后补入 PRD；低优先级可标注"不适用"后跳过。
- PRD 中必须消除实现歧义：数据字段是否必需、组件是否可复用、状态切换是否互斥、动画是否阻塞操作、外部系统是否属于核心功能范围。
- 原始框架坐标（如 Egret x/y/width/height）仅用于推断区块的相对位置关系，不得直接写入 HTML 定位；HTML 实现映射必须采用 flex / grid 等常见 Web 布局，核心交互逻辑的完整性优先于视觉布局的像素精度。
- 多场景模板必须显式区分 `page-scene`、`overlay-scene`、`substate`；不能只在 `ui.md` 或 `interaction.md` 中隐式散写场景信息。
- 若用户目标是“让 PRD 驱动新系统生成模板”，必须继续检查并补齐 `scenes.md`、`generation.md`、`prompt.md` 与共享能力文档，而不是停在基础分析 PRD。
- 若用户是在审查现有 PRD 的生成充分性，应优先从 `index.md` 和现有子文件建立判断，只在某个结论缺少证据时最小回源到源码。
- `index.md` 必须写清入口身份、读取顺序、文件职责和跳过风险；不能只做摘要页。
- PRD 超过 300 行或用户要求"按需载入"时，应输出多文件拆分格式（见"多文件拆分模式"章节），`index.md` 为每次必读入口，其余子文件按任务类型按需载入。
